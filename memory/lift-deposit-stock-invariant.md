A completed **lift** draws against supplier **deposits**: on create, `applyAmountAgainstSupplierDeposits(lift.total_amount)` moves money `balance_remaining → balance_used`; on edit/delete, `creditAmountBackToSupplierDeposits(lift.total_amount)` reverses it (pooled FIFO across the supplier's deposits, no per-lift tracking). **Invariant:** a supplier's total `balance_used` ≈ sum of their completed lift `total_amount`s. Any code that changes `lift.total_amount` MUST reconcile deposits by the delta, or edit/delete will mis-credit.

**The "this lift cannot be edited because some bottles have already been sold" error was a RED HERRING.** Old `revertCompletedLift`/`destroy` inferred "sold" from `current_purchased_quantity < number_of_cases*bottles_per_case`. The Product List **Adjust Inventory** feature (`adjustVariantStock`) lowers `current_purchased_quantity` with no sale, so any downward stock adjustment falsely tripped it. Fixed by detecting real sales via `liftItemHasSales()` — queries `sale_items` (non-draft `sale`, bottles_sold > 0) instead of the running balance. See [[stock-value-correction-procedure]].

**Editing a completed lift now works WITH partial sales (2026-06-06).** The old hard block is gone. Rule (per variant, confirmed by user): the new lifted quantity must be `>= already-sold bottles` (enforced on purchased AND free pools separately); reducing below sold, or removing a sold variant, rejects the WHOLE update (`ValidationException`, key `lift`, message names the variant). Increasing / price / free / cost edits are always allowed. **Critical mechanism:** the completed-lift edit path was switched from delete+recreate to an **in-place reconcile** (`LiftRepository::reconcileCompletedLiftItems`) that PRESERVES each `products.id` — otherwise `sale_items.product_id` would orphan against a soft-deleted batch and the running stock would break. Existing variants update in place (`current_purchased = lifted − sold`, `meta total_cost = remaining` if sold else full); new variants create fresh batches; removed (unsold) variants soft-delete. Deposit reconciled by `delta = newTotal − oldTotal` (no full credit-then-reapply). `LiftController::validateLiftEditAgainstSales` + `soldBottlesByProduct`/`soldBottlesForLiftItem` do the gate. `destroy()` still fully blocks deleting a lift with sales (uses `liftItemHasSales`). Shared metric helpers `computeVariantMetrics`/`buildMetadataVariant`/`buildLiftItemAttrs`/`createProductBatch` keep create + edit paths in sync. Verified via `scripts/test_lift_edit.php` (rolled-back scenarios).

After that, the true blocker surfaced: `creditAmountBackToSupplierDeposits` threw `RuntimeException('Unable to restore previous deposit usage...')` because `adjustVariantStock` had recomputed `lift.total_amount` (to update the Lift Report) WITHOUT touching deposits — so `total_amount` (e.g. 33333.34) exceeded recorded `balance_used` (e.g. 20000). Two-part fix:
1. `creditAmountBackToSupplierDeposits` now caps the credit at `sum(balance_used)` instead of throwing — self-heals on re-save (revert credits what was used, then re-apply sets it to the new total).
2. `adjustVariantStock` now reconciles deposits by `delta = newTotal - oldTotal` per affected lift (apply if +, credit if −; completed lifts only), wrapped in try/catch → `back()->withErrors` if the increase exceeds the deposit balance. **User chose "reconcile deposit immediately" (2026-06-06)**: stock adjustments are treated as purchase-value corrections that move supplier money.

Pre-fix desynced lifts (like Lift#3) self-heal the moment the user edits/saves them (verified: edit → deposit `balance_used` becomes the new lift total).
